# Agent Loop 原型 — 架构设计文档

> 版本：v1.0 | 日期：2026-05-12 | 状态：待 Claude Code 执行

---

## 一、项目定位

**从「传统 Flask 应用」转向「AI Agent 常驻智能体」。**

现有 v2 系统（`认知增强系统/`）是成熟的传统架构：Flask + Vue 3 + SQLite + 规则引擎。本原型在旁路（`agent_proto/`）搭建新一代 Agent 运行时，验证核心假设：

> 系统基于对用户的观察，自主判断时机，主动发起恰当的自然语言交互。

原型成功标准：Agent 自主说出一句恰当的话（终端输出即可），完成一次 Observer→Thinker→Actor 闭环。

## 二、核心架构：Observer → Thinker → Actor 循环

```
┌─────────────────────────────────────────────────┐
│                  Agent Runtime                    │
│                                                   │
│  ┌──────────┐    ┌──────────┐    ┌────────────┐ │
│  │ Observer │ ─→ │ Thinker  │ ─→ │   Actor    │ │
│  │ (采集)   │    │ (决策)   │    │  (行动)    │ │
│  │          │    │          │    │            │ │
│  │ 系统钩子  │    │ LLM调用  │    │ 终端输出    │ │
│  │ 定时器   │    │ 上下文   │    │ 文件写入    │ │
│  │ v2复用   │    │ 人格卡   │    │ (后续:通知) │ │
│  └──────────┘    └──────────┘    └────────────┘ │
│       ↑                               │          │
│       └──────── 反馈循环 ────────────┘          │
│                                                   │
│  ┌─────────────────────────────────────────────┐ │
│  │           共享记忆层 (只读v2 SQLite)          │ │
│  │  切片记录 │ 情绪日志 │ 任务状态 │ 设备日志    │ │
│  └─────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────┘
```

### 模块职责边界

| 层 | 模块 | 用AI还是确定性代码 | 理由 |
|----|------|-------------------|------|
| Observer | 窗口监控、空闲检测、时间上下文 | **确定性 Python** | 数据采集需100%可靠，不可让AI经手 |
| Observer | 剪贴板/URL/通知监听 | **确定性 Python** | 同上，但采集后的语义理解可交给AI |
| Thinker | 判断「该不该说话」 | **本地 LLM (Qwen3.5-9B)** | 需要常识推理，但涉及隐私数据不出本地 |
| Thinker | 判断「说什么」 | **本地 LLM + 人格卡** | 同上 |
| Thinker | 长期趋势/跨日模式发现 | **云端 API (DeepSeek V4 Pro)** | 需要强推理+长上下文，数据先脱敏 |
| Actor | 终端输出/文件写入 | **确定性 Python** | 执行层不可出错 |
| Actor | 后续：通知推送/悬浮窗更新 | **确定性 Python** | 同上 |
| Memory | 读取 v2 SQLite | **确定性 Python** | 数据完整性 |

### 核心原则
- **AI 做决策，传统做执行**
- **AI 可建议，传统可拒绝**
- **健康数据/窗口标题/剪贴板原文绝不离开本地**
- **v2 代码只读，不修改**

## 三、三 Persona 设计

三个 Persona 共享同一 Agent 运行时，但各有独立的人格卡（prompt）和触发策略：

| Persona | 角色 | 触发场景 | 首次 MVP |
|---------|------|---------|----------|
| **切片记录者** | 细心的记录者 | 窗口切换、长时间活动、久未记录 | ✅ MVP 首个 |
| **情感陪伴者** | 关切的陪伴者 | 情绪异常检测、趋势变化、该复盘时 | 后续 |
| **任务调度者** | 干练的调度者 | 排程冲突、低精力+高难度任务、Deadline | 后续 |

### Persona 发声权（按存在模式）

| 模式 | 切片记录者 | 情感陪伴者 | 任务调度者 |
|------|-----------|-----------|-----------|
| 休眠态（托盘） | 静默记录，仅重要提醒 | 仅 L4 紧急介入 | 仅截止时间前提醒 |
| 陪伴态（悬浮窗） | 定期提醒记录 | 颜色表达情绪状态 | 显示今日任务数 |
| 对话态（聊天窗） | 主动回顾今日切片 | 主动问候、深度对话 | 排程讨论、冲突提醒 |

### 自主级别（L0-L4）

| 级别 | 名称 | 行为 | 适用场景 |
|------|------|------|---------|
| L0 | 静默观察 | 只记录，不打扰 | 日常活动追踪 |
| L1 | 温和建议 | 通知：「要不要记录一下？」 | 检测到可能的切片时机 |
| L2 | 预告行动 | 「检测到情绪波动，10秒后自动记录...」 | 异常检测触发 |
| L3 | 自主执行 | 自动记录，事后告知 | 环境切换、定时切片 |
| L4 | 紧急介入 | 立即对话，不可延迟 | 危机检测（严重焦虑模式） |

## 四、观察范围 — 完整配置

### 第一层：系统级信号 ✅ 全开
| 采集项 | 方法 | 用途 |
|--------|------|------|
| 活跃窗口标题 | pywin32 `GetWindowText` | 判断活动类型，决定模式切换和提醒时机 |
| 进程名 | psutil `process_name` | 粗粒度兜底 |
| 空闲时长 | `GetLastInputInfo` | 专注/休息/离开判断 |
| 系统时间 | `datetime.now()` | 时段区分（上午/下午/深夜） |

### 第二层：应用级信号 ✅ 全开（全量落盘）
| 采集项 | 方法 | 粒度 | 安全措施 |
|--------|------|------|---------|
| 剪贴板文本 | `win32clipboard` 或 `pyperclip` | **全量落盘** | 数据库文件级加密（SQLCipher），定期清理（>N天自动删除），仅本地9B处理 |
| 浏览器 URL | 浏览器扩展或窗口标题解析 | **完整 URL** | 写入前剥离敏感 query param（token、access_key 等） |
| 系统通知 | `winrt` 或 `winsdk` | **完整通知文字** | 仅本地处理；MVP 阶段如遇技术阻力可暂缓 |

### 第三层：内容级信号
| 采集项 | 决策 | 说明 |
|--------|------|------|
| 键盘节奏 | ✅ 开 | 打字速度、删除键频率、停顿模式。不记录按键内容 |
| 屏幕截图 | ❌ MVP 不开 | 隐私风险最高，后续若开需内存处理不落盘 |
| 音频 | ❌ 暂不开 | 技术链路过长，9B 语音情绪识别未验证 |

### 第四层：生理数据
| 采集项 | 决策 | 说明 |
|--------|------|------|
| 小米手环10 NFC | ✅ 预留接口 | MVP 不实现，数据接口预留（心率、HRV、睡眠） |

## 五、存在形态设计

智能体有三种「存在密度」，随用户注意力状态自适应切换：

```
休眠态（托盘）  ←→  陪伴态（悬浮窗）  ←→  对话态（聊天窗）
几乎消失            余光可及              完全在场
```

### 休眠态 — 系统托盘
- 形态：任务栏右下角图标
- 交互：仅气泡通知（重要事项才弹出）
- 智能体做的事：静默观察记录，L4 紧急介入仍生效
- 用户锁定方式：托盘菜单可锁定模式

### 陪伴态 — 悬浮窗（默认）
- 形态：桌面悬浮小圆点，参考灵动岛思路
- 视觉语言：**生命感的陪伴物**（非工具仪表盘）
- 状态表达：
  - 颜色：蓝=正常 / 橙=有洞察 / 红=紧急
  - 呼吸节奏：慢=无事 / 快=有话要说
  - 可选轻量文字（如「已3h未记录」）
- 交互：点击展开为对话窗口（灵动岛式变形）
- 存在感：安静待在桌面角落，会呼吸、会变化

### 对话态 — 聊天窗口
- 形态：灵动岛式从悬浮窗展开
- 交互：自然语言对话，Agent 可主动发起
- 三 Persona 在此统一：同一个窗口，同一智能体的不同侧面
- 关闭后缩回悬浮窗

### 模式切换策略
- **自动检测 + 静默切换**：基于活跃窗口、空闲时长、时段自动判断
- **用户可锁定**：右键菜单/指令锁定当前模式
- **切换信号**：悬浮窗呼吸节奏变化作为非语言提示

## 六、Thinker 模型配置

```
三层模型架构：

thinker:
  local:                    # 隐私推理
    model: qwen3.5-9b (Q4_K_M)
    endpoint: http://localhost:8080/v1
    use_for:
      - 健康数据推理
      - 隐私敏感判断（窗口标题、剪贴板、URL）
      - 高频简单决策（是否提醒、说什么）
      
  cloud_primary:            # 云端主力
    model: deepseek-v4-pro
    use_for:
      - 长上下文趋势分析
      - 跨日模式发现
      - 匿名化后的聚合数据推理
      
  cloud_multimodal:         # 多模态兜底
    model: qwen3.6-plus
    use_for:
      - 图像理解
      - 文档分析
      - 需要多模态的场景
```

### Thinker 运行频率
- **事件触发**：窗口切换、长时间空闲、时段切换（主要）
- **兜底轮询**：每 5 分钟一次（确保不遗漏）
- **不在每次 Observer 采样后运行**（10 秒一次太频繁）

## 八、与 v2 系统的关系

```
v2 系统（认知增强系统/）          Agent 原型（agent_proto/）
─────────────────────────        ─────────────────────────
app.py          Flask 主控        → 不碰
database.py     数据库管理        → agent_memory.py 只读
monitor.py      窗口监控          → agent_loop.py 复用Observer逻辑
rule_engine.py  硬编码规则        → agent_thinker.py 替代（LLM推理）
desktop.py      桌面托盘          → 后续替换为灵动岛式悬浮窗
scheduler.py    贪心排程          → 保留为确定性执行层
auto_slice_generator.py 自动切片  → 增强为LLM语义标注
```

**渐进策略**：原型验证通过后，逐步将 v2 的 Web 界面替换为智能体对话窗口，v2 代码退化为执行层。

## 九、关键设计约束

1. **隐私守则**：窗口标题、剪贴板原文、URL、通知内容仅本地 9B 处理，不送云端
2. **原型无 UI**：MVP 阶段终端输出验证闭环 → 下一阶段再建悬浮窗
3. **只读 v2 数据**：agent_memory.py 通过 SQLite ATTACH 或直接读取 `../认知增强系统/data/`
4. **不阻塞主线程**：Agent Loop 在独立线程中运行，Thinker 调用 LLM 时异步
5. **错误容忍**：LLM 调用失败不崩溃，降级为静默观察
