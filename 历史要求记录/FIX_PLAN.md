## P0 🔴 冷却期与窗口过滤 Bug（必须修，否则 Agent 聒噪到不可用）

### 背景

proto_log 中 57 分钟产生 50+ 条消息，理论冷却期 10 分钟完全失效。同一秒内出现两条 speak、18 秒内连续 speak。用户会关闭 Agent。

---

### P0-1：冷却期检查从「单批次内」提升为「全局跨批次」

**文件**：`agent_proto/agent_loop.py`

**根因**：
`_last_speak_time` 的检查和更新可能在 `_process_events()` 的 for 循环里，但不同批次的 events 由不同次的 `_process_events()` 调用处理。如果两个窗口快速切换（Alt+Tab 两次），events 可能分属两次 buffer 读取 → 各自通过冷却检查。

**修复要求**：
1. 在 `AgentLoop` 类中添加实例变量 `self._last_speak_time`（初始值 0），取代任何局部变量或循环内的冷却检查
2. 在 `_process_events()` 中，**进入 for 循环之前**先检查 `now - self._last_speak_time < SPEAK_COOLDOWN`：
   - 如果冷却期内 → 整个批次的 speak 事件全部跳过（直接 return 或 continue 到下一批次）
   - 如果冷却期已过 → 进入 for 循环处理事件，但**在 Actor 实际执行 speak 之后**立即更新 `self._last_speak_time = time.time()`
3. 同一批次内只允许一次 speak——一旦某个事件触发 speak，批次内后续事件跳过
4. 添加日志：冷却期拦截时打印 `[Cooldown] 距上次发言 X 秒，跳过 Y 个事件`

**预期行为**：
- 一次 speak 后，接下来 10 分钟内无论来多少事件，Agent 绝对不开口
- 10 分钟后第一个符合条件的触发事件才会 speak

---

### P0-2：心跳事件必须检查窗口持续时间

**文件**：`agent_proto/agent_loop.py`（事件分发逻辑）或 `agent_proto/agent_thinker.py`（心跳事件处理路径）

**根因**：
心跳事件（heartbeat）的处理路径中，没有调用窗口时长过滤。日志中出现「刚才在资源管理器待了 24 秒，记一笔？」——这个 24 秒窗口本应在 Observer 或 Thinker 层被过滤掉。

**修复要求**：
1. 找到心跳事件的处理分支（可能在 `_process_events` 或 Thinker 的 `decide()` 中）
2. 确保心跳事件在决定 speak 之前，检查当前活跃窗口的持续时间是否 ≥ `MIN_WINDOW_DURATION`（当前为 120 秒）
3. 如果持续时间不足 2 分钟 → 降级为静默（`should_speak: false`），不产生对话
4. 同样检查应应用于所有事件类型（window_switch、idle_end、period_change），确保一致性

**预期行为**：
- 仅在某个窗口上停留 ≥ 2 分钟后，Agent 才可能就该窗口发起记录提醒
- 快速切换（Alt+Tab 浏览文件等）不再触发任何 speak

---

### P0-3：同一秒内重复 speak 的兜底防护

**文件**：`agent_proto/agent_actor.py`

**根因**：
极少数情况下（如线程竞争），两个 Thinker 决策可能在极短时间内先后到达 Actor。

**修复要求**：
1. 在 Actor 的 speak 方法中添加一个**最小间隔检查**（如 3 秒）：
   ```python
   if time.time() - self._last_actor_speak < 3.0:
       return  # 丢弃，不输出
   ```
2. 这个检查是 P0-1 的兜底，不是替代——主力防护在 P0-1

## P1 🟡 Recorder 人格卡增强（让 9B 说话不重复）

### 背景

当前 9B 只有 4 个话术模板循环使用，缺乏智能体感。根因是 `recorder.md` 人格卡太薄——只给了触发规则，没给话术变化示例和情境判断逻辑。

---

### P1-1：添加话术多样化示例

**文件**：`agent_proto/agent_personas/recorder.md`

**修复要求**：
在人格卡中新增一个 `## 话术风格` 章节，包含：

1. **语气定位**（3 句以内）：
   - 你是一个细心的记录伙伴，语气温和、简洁、不带压力
   - 提醒是邀请而非命令，用户随时可以拒绝

2. **至少 8 种话术模板**（覆盖不同情境）：

   | 情境 | 话术示例 |
   |------|---------|
   | 刚切换上下文 | 「从 A 换到 B 了，A 上的事要记下来吗？」 |
   | 长时间专注后 | 「在 A 上忙了好一阵了，趁还记得，记一笔？」 |
   | 时段切换 | 「到下午了，上午主要在做什么？帮你记一下」 |
   | 空闲后回归 | 「休息回来了，刚才休息前在做什么还记得吗？」 |
   | 检测到碎片化 | 「过去一小时切换了好几个窗口，要梳理一下这段时间吗？」 |
   | 每日首记 | 「早上好，今天刚开始。昨天有什么想记的吗？」 |
   | 长时间未记录 | 「好久没记录了，最近几个小时主要在忙什么？」 |
   | 用户刚手动记过 | 「刚才已经记过了。现在在 A 上，要补充什么吗？」 |

3. **变化规则**：
   - 连续两次提醒不能使用同一模板
   - 如果用户连续 3 次拒绝（回复「不」「不用」「忙」等），接下来 30 分钟内降低提醒频率（只在时段切换时提醒）
   - 根据时段（上午/下午/晚上）调整问候语气

---

### P1-2：添加情境判断逻辑

**文件**：`agent_proto/agent_personas/recorder.md`

在已有的上下文格式之后，新增 `## 决策逻辑` 章节：

```markdown
## 决策逻辑

收到上下文后，按以下顺序判断：

1. **是否为有意义的活动？**
   - 窗口标题含「资源管理器」「设置」「任务管理器」等系统工具 → 通常不提醒
   - 窗口标题含 IDE、文档、浏览器具体页面 → 值得记录
   - 进程名是游戏 → 可以提醒（娱乐也是生活）

2. **持续时长够吗？**
   - < 2 分钟 → 不提醒（快速浏览/切换）
   - 2-15 分钟 → 可选提醒
   - > 15 分钟 → 值得提醒

3. **与上次记录间隔多久？**
   - < 10 分钟 → 大概率不提醒（除非活动显著不同）
   - 10-60 分钟 → 正常判断
   - > 2 小时 → 更主动提醒

4. **用户当前状态适合被打扰吗？**
   - 刚恢复活动（idle_end）→ 缓冲 1 分钟再判断
   - 持续打字中 → 等待，不打断
   - 空闲 > 30 秒 → 合适时机

5. **输出 JSON 前自检**：
   - 我上次说了什么？（避免重复）
   - 我现在说的话有新的信息量吗？
   - 如果只是「你在用 X」而没有更多洞察，宁可不说话
```

## P2 🟢 L2 观察者补全（用户之前明确要求的功能）

### 背景

用户在观察范围决策中明确选择了：
- 剪贴板文本：全量记录（方案 A）
- 浏览器 URL：完整 URL（方案 A）
- 系统通知内容：开启

当前代码中仅剪贴板记录了长度，浏览器 URL 和系统通知均未实现。

---

### P2-1：剪贴板全量记录

**文件**：`agent_proto/agent_observer.py`

**修复要求**：
1. 在 Observer 的采样线程中，使用 `pyperclip.paste()` 获取剪贴板文本
2. **去重**：与前一次内容相同则跳过（避免重复记录）
3. **长度限制**：单次记录上限 5000 字符（防止大文件复制撑爆内存）
4. **写入事件**：生成事件类型 `clipboard_change`，携带 `content`（文本内容）和 `length`（字符数）
5. **安全标记**：事件中标记 `sensitive: true`，Thinker 层看到此标记时必须走本地 9B，严禁发送到云端 API
6. **不落盘原则**：剪贴板内容在当前原型阶段通过事件队列传递，不写入 SQLite（后续阶段再加持久化）

**隐私注意**：
- 剪贴板内容仅通过内存中的事件队列传递
- Thinker 处理时标记 `sensitive: true`，确保只在本地 9B 推理
- Actor 对话中如引用剪贴板内容，需脱敏（如只提「你刚才复制了一段代码」而不展示代码原文）

---

### P2-2：浏览器 URL 监听

**文件**：`agent_proto/agent_observer.py`

**修复要求**：
1. 当活跃窗口进程为浏览器（chrome.exe, msedge.exe, firefox.exe）时，尝试获取当前标签页 URL
2. **实现方式**（按优先级尝试）：
   - Windows：通过 `pywin32` 获取浏览器窗口标题 → 解析 URL（Chrome/Edge 通常在标题中显示域名+页面名）
   - 备选：读取浏览器当前标签页的 `window.title` 中 URL 信息
   - 如果无法可靠获取完整 URL，先实现「从窗口标题提取域名」作为降级方案
3. **URL 清洗**：写入事件前剥离敏感 query 参数（token, access_key, session, auth, password 等）
4. **写入事件**：生成事件类型 `url_change`，携带 `url`（清洗后的完整 URL）和 `domain`（域名）
5. **去重**：同一域名 + 同一路径第一段连续出现时，每 5 分钟最多记录一次

---

### P2-3：系统通知监听

**文件**：`agent_proto/agent_observer.py`

**修复要求**：
1. 尝试通过 Windows 通知 API 监听系统通知
2. **实现路径**：
   - 方案 A：使用 `winrt` 库订阅 `UserNotificationListener`（需要 Windows 10/11 并在设置中授予权限）
   - 方案 B：如果方案 A 权限获取复杂，先实现占位接口 + 日志提示「通知监听需用户手动授权」
3. **当前阶段目标**：先实现方案 A 的探测代码，验证是否可用；如果权限不足则输出清晰说明，预留接口供后续手动授权后启用
4. **写入事件**：生成事件类型 `system_notification`，携带 `app_name`、`title`、`content`
5. **隐私标记**：`sensitive: true`，同剪贴板
